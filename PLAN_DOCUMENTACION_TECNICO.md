## 1. Descripción del Proyecto

### ¿Qué es SweetBox?

**SweetBox** es una aplicación móvil multiplataforma (iOS y Android) para la compra y entrega de postres artesanales personalizados. Permite a los usuarios explorar un catálogo de productos (cupcakes, pasteles, galletas, brownies, macarons), personalizar sus pedidos (sabor, relleno, mensaje), pagar de forma segura y rastrear su entrega en tiempo real.

### Problema que Resuelve

| Problema Actual | Solución SweetBox |
|-----------------|-------------------|
| Pedidos por WhatsApp sin confirmación automática | Plataforma estructurada con carrito y confirmación |
| No hay seguimiento del pedido | Rastreo GPS en tiempo real |
| Sin personalización digital | Opciones de sabor, relleno y mensaje por producto |
| Pagos informales (efectivo / transferencia manual) | Pago integrado con tarjeta / MercadoPago |
| Sin historial de compras | Perfil con historial y pedidos guardados |

### Modelo de Negocio

- **Comisión por venta**: El negocio retiene el 100% al ser una app propia
- **Futuro**: Modelo marketplace para múltiples reposterías (comisión 10–15%)

## 2. Objetivos

### Objetivo General
Desarrollar una aplicación móvil que digitalice el proceso de venta, personalización y entrega de postres artesanales para SweetBox, mejorando la experiencia del cliente y la eficiencia operativa del negocio.

### Objetivos SMART

| # | Objetivo | Indicador | Meta | Plazo |
|---|----------|-----------|------|-------|
| O1 | Lanzar MVP funcional | App publicada en stores | App Store + Play Store | 4 meses |
| O2 | Adquirir primeros usuarios | Descargas | 500 en el primer mes | Mes 5 |
| O3 | Generar ventas digitales | Pedidos procesados | 50 pedidos/semana | Mes 6 |
| O4 | Reducir tiempo de gestión | Tiempo por pedido | -40% vs. WhatsApp | Mes 5 |
| O5 | Satisfacción del cliente | Calificación promedio | ≥ 4.5/5 estrellas | Mes 6 |

## 3. Público Objetivo

### Segmento Primario — Comprador Final (B2C)

```
Perfil:
  - Edad: 20–40 años
  - Género: 60% femenino, 40% masculino
  - Ubicación: Zona urbana con cobertura de entrega
  - Dispositivo: Smartphone (iOS o Android)
  - Ocasiones: Cumpleaños, bodas, baby shower, aniversarios
  - Comportamiento: Compra online, valora la personalización
```

### Segmento Secundario — Administrador / Vendedor (B2B interno)

```
Perfil:
  - Dueño o empleado de SweetBox
  - Gestiona pedidos, catálogo e inventario
  - Necesita panel de control simple y eficiente
```

> **María, 28 años, Ciudad de México**  
> Trabaja en oficina, busca regalos únicos para eventos especiales.

| `POST` | `/addReview` | Agregar reseña de un pedido |

### Reglas de Seguridad de Firestore

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Usuarios: solo el propio usuario puede leer/escribir
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }

    // Productos: cualquier usuario autenticado puede leer
    match /products/{productId} {
      allow read: if request.auth != null;
      allow write: if request.auth != null && request.auth.token.admin == true;
    }

    // Pedidos: el usuario solo ve sus pedidos
    match /orders/{orderId} {
      allow read: if request.auth != null &&
        (resource.data.userId == request.auth.uid ||
         request.auth.token.admin == true);
      allow create: if request.auth != null;
      allow update: if request.auth != null && request.auth.token.admin == true;
    }

    // Reseñas: autenticado puede leer, solo propietario puede escribir
    match /reviews/{reviewId} {
      allow read: if request.auth != null;
      allow create: if request.auth != null &&
        request.resource.data.userId == request.auth.uid;
    }
  }
}
```

## 11. Seguridad

### Autenticación
- JWT tokens manejados por Firebase Auth (renovación automática)
- Tokens de acceso con expiración de 1 hora
- Refresh tokens almacenados de forma segura (Expo SecureStore)

### Datos Sensibles
- ❌ Nunca almacenar datos de tarjeta en la app ni en Firestore
- ✅ Tokenización de pagos via Stripe (PCI DSS Compliant)
- ✅ HTTPS obligatorio en todas las comunicaciones
- ✅ Variables de entorno para API keys (no hardcoded)

### Protección de la App
```
- Expo SecureStore: almacenar tokens y datos sensibles del dispositivo
- Certificate Pinning: para comunicaciones con backend crítico
- Obfuscación de código en build de producción
- Rate limiting en Firebase Functions
```

## 13. Criterios de Aceptación

### Criterios Globales (todos los sprints)

- [ ] La funcionalidad pasa las pruebas unitarias (cobertura ≥ 80%)
- [ ] La UI es responsiva y funciona en pantallas de 5" a 6.7"
- [ ] El tiempo de carga de pantalla es < 2 segundos
- [ ] No hay errores de lint ni advertencias críticas
- [ ] El código está revisado por al menos un developer (code review)
- [ ] La funcionalidad es accesible (texto legible, contraste WCAG AA)

### Criterios por Módulo

#### Auth
- [ ] El usuario puede registrarse con email y contraseña
- [ ] El usuario puede iniciar sesión con Facebook
- [ ] El usuario puede recuperar su contraseña por email
- [ ] Los errores de validación se muestran claramente

#### Catálogo
- [ ] Los productos cargan en < 1.5 segundos
- [ ] Los filtros actualizan el listado sin recargar pantalla
- [ ] La búsqueda muestra resultados al escribir (debounce 300ms)
- [ ] Las imágenes cargan con lazy loading y placeholder

#### Carrito
- [ ] El usuario puede personalizar cada producto (sabor, relleno, mensaje)
- [ ] El total se actualiza automáticamente al cambiar cantidades
- [ ] El carrito persiste si el usuario cierra la app

#### Pagos
- [ ] El pago se procesa correctamente con tarjeta de prueba Stripe
- [ ] El usuario recibe confirmación visual y por push notification
- [ ] Los errores de pago se muestran con mensaje claro

#### Rastreo
- [ ] El mapa muestra la ubicación del repartidor actualizada cada 10 segundos
- [ ] El estado del pedido cambia en tiempo real
- [ ] La estimación de tiempo es visible

## 15. Riesgos y Mitigación

| # | Riesgo | Probabilidad | Impacto | Mitigación |
|---|--------|-------------|---------|-----------|
| R1 | Rechazo de la app en App Store (política de Apple) | Media | Alto | Revisar App Store Guidelines antes de desarrollar |

| R4 | Retrasos en el cronograma | Media | Alto | Buffer de 1 semana en Sprint 7 (QA) |
| R5 | Escalabilidad de Firestore | Baja | Alto | Diseñar índices correctamente desde el inicio |
| R6 | Problemas de UX detectados tarde | Media | Medio | Testing con usuarios reales desde el Sprint 2 |
| R7 | Cambios de requisitos del cliente | Alta | Medio | Scrum permite ajustes en cada sprint |
| R8 | Presupuesto insuficiente | Baja | Alto | Priorizar MVP y dejar features secundarios para Fase 2 |
